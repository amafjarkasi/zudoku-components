Understanding how Zephyr's platform is designed at a high level will help you build more resilient integrations and troubleshoot issues faster.

## Overview

Our platform is built on a microservices architecture deployed across multiple availability zones. Every public API you interact with is fronted by an intelligent API Gateway that handles authentication, rate limiting, request routing, and observability — before a single line of your business logic is executed.

## API Gateway

The API Gateway is the single entry point for all external traffic. It provides:

| Capability | Detail |
|---|---|
| **TLS Termination** | TLS 1.2+ enforced; 1.3 preferred |
| **Authentication** | JWT validation + API key lookup |
| **Rate Limiting** | Token bucket, per-key and per-IP |
| **WAF** | OWASP Top 10 rule set, custom rules |
| **Observability** | P50/P95/P99 latency per route |
| **Circuit Breaker** | Automatic upstream failure isolation |

## Regional Deployment

Our services are deployed in three primary regions with automatic failover:

| Region | Location | Latency Target |
|---|---|---|
| `us-east-1` | N. Virginia | Primary / `< 25 ms` |
| `eu-west-1` | Ireland | EU Primary / `< 30 ms` |
| `ap-southeast-1` | Singapore | APAC Primary / `< 40 ms` |

All API requests are routed to the nearest healthy region via Anycast DNS. You don't need to manage region selection — it's handled automatically.

<Callout type="info">
  **Geo-pinning**: If your compliance requirements mandate data residency in a specific region, contact your account team to configure geo-pinned routing.
</Callout>

## Data Flow

### Synchronous Requests

Standard API calls (GET, POST, PUT, DELETE) follow a synchronous request/response pattern. Responses are returned directly from the service within the same HTTP connection.

<Mermaid chart={`

    participant D as Database
    C->>G: API Request
    G->>S: Routed Request
    S->>D: Query
    D-->>S: Data
    S-->>G: JSON Response
    G-->>C: Response
`} />

**Typical P95 latency**: `< 150ms` end-to-end.

### Asynchronous Events

Long-running operations (order fulfillment, report generation, bulk imports) are processed asynchronously:

1. Your API call returns a `202 Accepted` with a `jobId`
2. The service publishes an event to the Kafka event bus
3. A downstream worker processes the job
4. On completion, a webhook fires to your registered endpoint
5. You can also poll `GET /jobs/{jobId}` for status

```json
// 202 Accepted response
{
  "jobId": "job_01HX2K4M9Q8NVRP3",
  "status": "queued",
  "estimatedCompletionMs": 3500,
  "statusUrl": "https://api.zephyr.com/v1/jobs/job_01HX2K4M9Q8NVRP3"
}
```

## Service Communication

Internal service-to-service communication uses gRPC with mTLS. This means:

- **No extra latency** from repeated TLS handshakes between services
- **Type-safe contracts** enforced by Protocol Buffers
- **Automatic service discovery** via Consul

From your perspective as an API consumer, this is completely transparent — you interact only with the public REST/HTTP API surface.

## Caching Strategy

| Layer | Technology | TTL |
|---|---|---|
| DNS | Anycast | 60 s |
| CDN (read-only endpoints) | Cloudflare | 30 s – 5 min |
| Application cache | Redis Cluster | Varies per endpoint |
| DB query cache | PgBouncer | Per query plan |

<Callout type="info">
  Cached responses include a `X-Cache-Status: HIT` header. If you're debugging stale data, check this header first and use `Cache-Control: no-cache` to bypass CDN caching in your requests.
</Callout>

## SDK Architecture

Our official SDKs wrap the HTTP API with:

- **Automatic retry** with exponential backoff (3 attempts by default)
- **Connection pooling** to reduce handshake overhead
- **Type-safe response models** generated from OpenAPI specs
- **Telemetry hooks** for custom observability integration

See the [React SDK](/sdks/react) and [Python SDK](/sdks/python) guides for setup instructions.
