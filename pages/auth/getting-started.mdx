Security is our top priority. The Authentication API provides the tools you need to secure your integrations, whether you are building a simple script or a complex multi-tenant SaaS application.

## Authentication Methods

We support multiple authentication methods designed for different use cases:

| Method | Best For | Security Level | Complexity |
|--------|----------|----------------|------------|
| **API Keys** | Server-to-server, cron jobs, scripts | High | Low |
| **OAuth 2.0** | User-facing applications, third-party integrations | Very High | Medium |
| **JWT Tokens** | Stateless authentication, microservices | High | Medium |

### API Keys

**Best for:** Server-side applications, backend services, CLI tools, and scripts where you control the environment.

**Advantages:**
- Simple to implement
- No user interaction required
- Easy to rotate and revoke
- Fine-grained permissions per key

**Disadvantages:**
- Not suitable for client-side applications (browsers, mobile apps)
- Must be kept secret

### OAuth 2.0

**Best for:** Web applications, mobile apps, and any application where users need to authenticate with their own Zephyr account.

**Advantages:**
- Industry-standard protocol
- Users grant specific permissions (scopes)
- Built-in token refresh mechanism
- Revocable by users

**Disadvantages:**
- More complex implementation
- Requires redirect handling
- User interaction required

## Getting an API Key

API keys are the simplest way to get started. Follow these steps to generate your first key:

### Step 1: Access Your Dashboard

1. Log in to [dashboard.zephyr.com](https://dashboard.zephyr.com)
2. Navigate to **Settings > API Keys**

### Step 2: Generate a New Key

1. Click the **Generate New Key** button
2. Fill in the key details:

| Field | Description | Example |
|-------|-------------|---------|
| **Name** | Descriptive name for the key | "Production Backend" |
| **Permissions** | Scopes this key can access | `products:read`, `orders:write` |
| **IP Whitelist** | Optional CIDR blocks that can use this key | `10.0.0.0/8, 192.168.1.0/24` |
| **Expiration** | Optional expiration date | "2026-12-31" |
| **Rate Limit** | Custom rate limit for this key | 1000 req/min |

3. Click **Generate Key**

<Callout type="info">
  **Important:** Copy your API key immediately after generation. You won't be able to see it again for security reasons!
</Callout>

### Step 3: Store Your Key Securely

**Good practices:**
- Store keys in environment variables or secret management systems
- Never commit keys to version control
- Use different keys for different environments (dev, staging, prod)
- Rotate keys regularly (we recommend every 90 days)

**Example - Environment Variables:**

```bash
# .env file (never commit this!)
ZEPHYR_API_KEY=sk_live_abc123xyz789def456...
```

```javascript
// Load environment variables
import dotenv from 'dotenv';
dotenv.config();

const apiKey = process.env.ZEPHYR_API_KEY;
```

<Callout type="info">
  **Never commit your API keys to version control!** Always use environment variables (e.g., `.env` files) to store sensitive credentials. If we detect a compromised key on public GitHub, it will be automatically revoked.
</Callout>

## Using API Keys

### Basic Usage

Include your API key in the `Authorization` header with the `Bearer` scheme:

```bash
curl -X GET https://api.zephyr.com/v1/products \
  -H "Authorization: Bearer sk_live_abc123xyz789def456..." \
  -H "Content-Type: application/json"
```

### Using SDKs

Our SDKs handle authentication automatically:

```javascript
// Node.js
import { ZephyrClient } from '@zephyr/node-sdk';

const client = new ZephyrClient({
  apiKey: process.env.ZEPHYR_API_KEY
});

const products = await client.products.list();
```

```python
# Python
from zephyr import ZephyrClient

client = ZephyrClient(api_key='sk_live_abc123xyz789...')
products = client.products.list()
```

### API Key Permissions

API keys support granular permissions to follow the principle of least privilege:

| Permission | Description | Risk Level |
|------------|-------------|------------|
| `products:read` | Read product information | Low |
| `products:write` | Create/update/delete products | Medium |
| `orders:read` | Read order information | Medium |
| `orders:write` | Create/update orders | High |
| `customers:read` | Read customer data | Medium |
| `customers:write` | Modify customer data | High |
| `analytics:read` | Access analytics data | Low |
| `webhooks:write` | Manage webhooks | Medium |
| `admin:all` | Full administrative access | Critical |

<Callout type="info">
  **Security Warning:** Only grant the minimum permissions needed. Avoid using `admin:all` unless absolutely necessary.
</Callout>

**Example - Creating a restricted key for a read-only service:**

```bash
curl -X POST https://api.zephyr.com/v1/api-keys \
  -H "Authorization: Bearer sk_live_abc123..." \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Analytics Dashboard Service",
    "permissions": [
      "products:read",
      "orders:read",
      "analytics:read"
    ],
    "ip_whitelist": ["10.0.0.0/24"],
    "rate_limit": 500
  }'
```

### Managing API Keys

#### List Your Keys

```bash
curl -X GET https://api.zephyr.com/v1/api-keys \
  -H "Authorization: Bearer sk_live_abc123..."
```

**Response:**

```json
{
  "data": [
    {
      "id": "key_abc123",
      "name": "Production Backend",
      "key_prefix": "sk_live_abc123...",
      "permissions": ["products:read", "orders:write"],
      "ip_whitelist": ["10.0.0.0/8"],
      "created_at": "2026-05-01T00:00:00Z",
      "last_used_at": "2026-05-10T14:30:00Z",
      "expires_at": "2026-12-31T23:59:59Z",
      "status": "active"
    }
  ]
}
```

#### Revoke a Key

```bash
curl -X DELETE https://api.zephyr.com/v1/api-keys/key_abc123 \
  -H "Authorization: Bearer sk_live_abc123..."
```

#### Rotate a Key

When you need to replace a key without downtime:

1. **Generate a new key** with the same permissions
2. **Update your application** to use the new key
3. **Test thoroughly** in production
4. **Revoke the old key** once confirmed working

```bash
# Step 1: Generate new key
curl -X POST https://api.zephyr.com/v1/api-keys \
  -H "Authorization: Bearer sk_live_abc123..." \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Production Backend V2",
    "permissions": ["products:read", "orders:write"],
    "ip_whitelist": ["10.0.0.0/8"]
  }'

# Step 4: Revoke old key (after testing)
curl -X DELETE https://api.zephyr.com/v1/api-keys/key_old_abc123 \
  -H "Authorization: Bearer sk_live_abc123..."
```

## Error Handling

### Common API Key Errors

#### Invalid API Key

```json
{
  "error": {
    "type": "authentication_error",
    "message": "Invalid API key provided",
    "code": "invalid_api_key",
    "request_id": "req_abc123xyz",
    "timestamp": "2026-05-10T15:30:00Z"
  }
}
```

**Cause:** The API key is malformed, expired, or doesn't exist.

**Solution:** Verify your API key is correct and hasn't been revoked.

#### Insufficient Permissions

```json
{
  "error": {
    "type": "authorization_error",
    "message": "Your API key does not have permission to perform this action",
    "code": "insufficient_permissions",
    "details": {
      "required": "orders:write",
      "granted": ["orders:read"]
    },
    "request_id": "req_abc123xyz",
    "timestamp": "2026-05-10T15:30:00Z"
  }
}
```

**Cause:** The API key lacks the required permission for the requested action.

**Solution:** Grant the required permission to your API key or use a different key with appropriate permissions.

#### IP Not Allowed

```json
{
  "error": {
    "type": "authorization_error",
    "message": "Your IP address is not whitelisted for this API key",
    "code": "ip_not_allowed",
    "details": {
      "your_ip": "192.168.1.100",
      "whitelisted_cidrs": ["10.0.0.0/8"]
    },
    "request_id": "req_abc123xyz",
    "timestamp": "2026-05-10T15:30:00Z"
  }
}
```

**Cause:** Your IP address is not in the key's whitelist.

**Solution:** Add your IP address to the whitelist or remove the IP restriction.

#### Rate Limit Exceeded

```json
{
  "error": {
    "type": "rate_limit_error",
    "message": "Rate limit exceeded",
    "code": "rate_limit_exceeded",
    "details": {
      "limit": 100,
      "remaining": 0,
      "reset_at": "2026-05-10T15:31:00Z"
    },
    "request_id": "req_abc123xyz",
    "timestamp": "2026-05-10T15:30:00Z"
  }
}
```

**Cause:** You've exceeded the rate limit for this API key.

**Solution:** Implement exponential backoff and retry after the `reset_at` time. See our [Rate Limiting Guide](/advanced/rate-limiting) for best practices.

## Security Best Practices

### 1. Never Expose API Keys in Client-Side Code

❌ **Don't do this:**

```javascript
// This is visible to anyone in the browser!
const apiKey = 'sk_live_abc123...';

fetch('https://api.zephyr.com/v1/products', {
  headers: {
    'Authorization': `Bearer ${apiKey}`
  }
});
```

✅ **Do this instead:**

```javascript
// Server-side only
app.get('/api/products', async (req, res) => {
  const response = await fetch('https://api.zephyr.com/v1/products', {
    headers: {
      'Authorization': `Bearer ${process.env.ZEPHYR_API_KEY}`
    }
  });
  const data = await response.json();
  res.json(data);
});
```

### 2. Use Separate Keys for Different Environments

```bash
# Development
ZEPHYR_DEV_KEY=sk_test_abc123...

# Staging
ZEPHYR_STAGING_KEY=sk_test_def456...

# Production
ZEPHYR_PROD_KEY=sk_live_ghi789...
```

### 3. Implement Key Rotation

Set up a reminder system to rotate keys every 90 days:

```javascript
// Check key age and alert if > 90 days
const keyCreationDate = new Date('2026-02-10');
const daysSinceCreation = (Date.now() - keyCreationDate) / (1000 * 60 * 60 * 24);

if (daysSinceCreation > 90) {
  console.warn('⚠️  API key is more than 90 days old. Consider rotating it.');
}
```

### 4. Monitor API Key Usage

Use the dashboard to monitor key usage patterns:

```bash
# Get usage statistics for a specific key
curl -X GET https://api.zephyr.com/v1/api-keys/key_abc123/usage \
  -H "Authorization: Bearer sk_live_abc123..." \
  -d '{
    "from": "2026-05-01T00:00:00Z",
    "to": "2026-05-10T23:59:59Z"
  }'
```

### 5. Use IP Whitelisting

Restrict API key usage to specific IP ranges:

```bash
curl -X POST https://api.zephyr.com/v1/api-keys/key_abc123 \
  -H "Authorization: Bearer sk_live_abc123..." \
  -H "Content-Type: application/json" \
  -d '{
    "ip_whitelist": [
      "10.0.0.0/8",      # Your VPC
      "192.168.1.0/24",  # Your office network
      "203.0.113.5/32"   # Specific server IP
    ]
  }'
```

### 6. Implement Webhook Notifications for Key Events

Get notified when API keys are created, rotated, or revoked:

```bash
curl -X POST https://api.zephyr.com/v1/webhooks \
  -H "Authorization: Bearer sk_live_abc123..." \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://your-domain.com/webhooks/api-keys",
    "events": ["api_key.created", "api_key.rotated", "api_key.revoked"],
    "secret": "whsec_abc123..."
  }'
```
